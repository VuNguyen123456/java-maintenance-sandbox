Maintainability Assessment

What made the original program harder to change
- The same rounding logic was duplicated inside many conversion methods, so any update to rounding/accuracy required editing many places and risked missing one. This increases maintenance effort and the chance of inconsistent behavior.
- UI behavior (printing/formatting) and conversion logic were tightly coupled, which encouraged copy‑paste edits when adding features.

What made the original program easier to change
- Each conversion already had a dedicated method, and the menu/dispatch structure was simple, so it was straightforward to add new options (like time conversions) and connect them to the UI.

What we changed to improve maintainability
- We centralized parsing, rounding, and formatting in shared helper methods, and made each conversion method only define its formula. This means changing accuracy/rounding now happens in one place.
- We made decimal accuracy a configurable setting (0–4) that the UI sets once and the rest of the program uses automatically.

What we would do differently if we did it again
- Separate UI from conversion logic earlier (e.g., a ConversionService class plus a CLI class) and add a small set of unit tests for each conversion + rounding edge cases (negatives, very small values, accuracy=0). This would make future changes safer and faster.
